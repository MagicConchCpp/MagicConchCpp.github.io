今天在解答一个前置递增和后置递增的问题时，发现了一个很有趣的问题：为什么不叫 ++C ，而是叫 C++ 呢？ 我被这个问题惊艳到了，没想这里还蕴含着编程知识。叫做 C++ 是因为后置递增会创建一个临时量以供返回原值，然后再进行自增。这说明 C++ 保留了 C，我们也知道 C++ 一直都几乎兼容 C 的。

这个临时量又引出另一各问题：使用前置递增还是后置递增呢？。最好的建议是，除非必须使用前置递增，否则选用后置递增。前置递增浪费了一些空间，当我们使用指针、整型（4字节）的自增时，会浪费很大空间。

今天还查阅了一下著名书籍《C++程序设计语言》（The C++ Programming Language），其实为什么不叫 ++C ，而是叫 C++ 呢？ 这个问题就可在其中查询到。书很厚，中文版大概1100页，C++Primer 才不到900页吧。没有系统读过，但是应该很优秀，唯一的缺点是书中代码的字体让我不太喜欢。

今天学到了移动操作 std::move ，还没学完，不过目前还是一脸懵逼。当我们赋值时，例如 a = b，我们首先会 构造出一个 b 的副本，然后将这个副本赋值给 a ，然后在销毁这个副本，我们为了避免这分配副本和销毁副本的开销，引入了 移动操作。这本书里面对于 移动操作 的引出给了一个很好地例子。当我借车给你用时，我会直接把车钥匙给你，而不是将我的车复制一个。

  2. << 运算符重载函数

    1. 为什么这个函数，不写成类的成员函数呢？
      如果写成成员函数，那么 << 就会作用于它左侧的对象，我们会这么使用这个函数 c1 << cout，这违背了我们的习惯。

    2. 为什么返回值是 ostream& ？
      如果我们仅仅输出一个值，cout << c1，我们完全可以返回 void。但有时我们会输出多个值，cout << c1 << c2，这时执行完 cout << c1 后，还要想输出 c2，就要保证 cout << c1 返回的还是和 cout 一样的类型（ostream&）。

2019年12月

2019-12-31

用别人现成的东西的好处就是，简单快捷，甚至不需要动脑就能解决问题。就比如直接用hexo 或者 WordPress写博客一样，你甚至不需要了解任何一点前端知识。
但是，作者提供的功能毕竟只是主流的，每个人都有自己的审美观，甚至于死抠一些非常细节的东西。要知道，作者是不能满足每个人的。
当这个作品不能满足你时，这时我们才意识到，如果我会自己开发就好了，所有的细节都由自己掌控就好了。虽然大部分时候，我们会放弃细节而选择效率。

昨日，在使用Markdown中的插入LaTeX公式的功能时，本地预览正常，但网站始终渲染不出公式。查阅资料得知，这是hexo自动将 \\ 和 \_ 认定为转义字符， 这和LaTeX语法冲突。
本来想既然你转义，我就再给你套一个转义，\\ 改为 \\\\ ， \_ 改为 \\_ 。在网站上倒是正常了，本地预览则是语法错误。

另外试了很多方法：

  1. 修改marked源文件 全部取消转义， 网上的代码已经过时， 我也不知道该改哪一处。
  2. 更改渲染源为karma， 不过Icarus主题貌似不支持。
  3. 修改js文件， 使当其遇到 $…$ 语句时取消转义， 但是因为主题不同，代码不起作用。
  4. 最后妥协了， 用hexo自带的一种“标签”环境，将公式放到 ... 中，hexo不会对标签环境的字符进行转义。唯一瑕疵就是本地markdown不认识这个“标签”环境，预览时多了那两行不算碍眼的字。
    算是，勉强解决吧。

今日，跨年之日，又陷入了很尴尬的境地-markdown目录问题。

icuras主题自带侧边目录功能，但其目录不能在文内展示，且和插件 hexo-toc 有冲突，不能同时存在(这就让人很纳闷)。
为了保留侧边目录，并且在文内也保留目录，且都具有跳转功能。（这个想法似乎很多余，但是为了使文章能够在摘要里显示目录，这是必须的）。
为此，我也试遍了许多方法。hexo-toc, doctoc, vscode插件里的各种toc，最后的结果都不尽人意。
按F12对比网站正常的跳转和非正常跳转的目录之间的区别，发现，两者的超链接是有细微差别的，对于标点字符和中文字符的处理不太相同。现在的目标是，将错误的如何改成正确的。但是，没有发现能够修改目录格式的插件。就是那么一点细微的处理差别，导致不能使用，也是很令人头疼。

本想，直接写个c++代码处理一下文档，生成个正确格式的目录（按理说Python做这种事更简单）。
但是，每次都要改改代码中文件名似乎很麻烦，于是想着自己开发个VSCode插件，一个快捷键就生成了，岂不美哉！借用同性恋者Brad的一句话：I’ll try anything once。（我是直的！）
人生苦短，何妨一试呢？
